## Key points
- **Three layers** identified: (1) catalog data model (product / component / component-in-product), (2) the intermediate proxy/technical data model in CREASHO that converts catalog data to Billing format, (3) business scenarios/edge cases. This call targeted layer 3.
- Any catalog change goes through a **revision** (logged unit of change, some requiring approval). A master process lists ~20–30 event types (create/publish, product changes, component changes, activation/deactivation, add/remove component, add datacenter, price changes, contract-type change, etc.). A "notification to billing" column marks which 16 are synced.
- Billing receives 100% of the fields Zhenya requested; only fields that change are sent. Some fields are constants hardcoded on the CREASHO side (e.g. always English for a "FieldNameRussian" field; no non-standard PriceType), some are computed (e.g. `show on customer portal` derived from `self-service order available`). The truth of this conversion logic lives in the billing-dedicated process.
- **Data model note**: product has many-to-many components; components have one-to-many prices. Prices are logically transactional — editing is forbidden; a new price record is created (e.g. to add USD, all other currencies are pulled in and USD added). Billing has no separate price-list concept — a single-currency change means archive whole product + create new one; ~6 prices × currencies × tax variants become multiple Billing "products."
- **Price change scenarios** to document: extending price validity dates; zeroing a price already sent to Billing; changing a price already sent; adding a currency not yet in Billing; standard vs. price-upon-request; quote-only prices.
- **Repricing existing customers**: handled via a "price changes" request; requires exporting affected customers from Billing (VOC), collaborative Excel, Growth splits list into "never reprice / call individually / reprice," marketing emails the repricing group. Contractual/relationship notification timelines apply (~1 month; date math around calendar month-ends — e.g. reprice on 29th → +next month, on 15th → 45 days). Billing gets two emails (standard + custom instructions) telling it to follow the special flow, not the default archive-all.
- **GAF / government-compliance products**: child products auto-generated from parents in the catalog with full field duplication except prices (multiplied by a coefficient). No separate revisions created for GAF children, but their data must still be pushed. Open question whether Billing should understand a parent/child concept or treat GAF as a black box (CREASHO stays master).
- **Vendors/distributors**: each component has exactly one vendor per region; the same product from different distributors (e.g. Microsoft via different suppliers) must be created as **separate components** in CRM, hence separate Billing products.
- **Naming**: biggest legacy pain was identical product naming across CRM and Billing. They cannot be literally identical (one CRM product → ~15 Billing products with price-list/currency/VAT suffixes), but roots match modulo human factor. A backlog of mass sync remains (~½–⅔ done over a year); field `billing sync status` (~90% reliable) shows added/not-added/synced/discrepancy.
- Category is the one reference field sent to Billing "as-is"; renaming/deleting a category is not currently handled and would break on next product change (inferred as a minor, unhandled edge case).
- Current integration is semi-manual (Excel files emailed → Billing manually processes via Jira); 100% accuracy is not guaranteed given change frequency.
