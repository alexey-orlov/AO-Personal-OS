## TL;DR
- SoftServe's Alex Orlov delivered an enablement session for Conga on building an AI-native "operating system" for product managers, framed around context engineering, skills, and sub-agents.
- Core thesis: 60% of agentic-AI success comes from designing and maintaining context (knowledge, registries, templates); model choice and prompting matter far less.
- Live demo drafted a PRD inside VS Code Agent Mode using a repository of ~80 PM skills and ~10 sub-agents; full PRD review by virtual reviewers didn't finish in-session.

## Key points
- Audience poll: most PM teams at Conga are at stage A — ad hoc, early-pioneer AI adoption with no shared agents or governance.
- Go-to-market/sales already uses AI heavily (research, proposals, decks) but it's tribal, ungoverned, and not connected to the PM org.
- Two agentic-framework archetypes contrasted: structured no-code (e.g., Copilot Studio) vs. Git-like file repository (VS Code + agent mode); session focused on the repo archetype.
- Building blocks introduced:
  - **Instructions**: system prompt (~500-line root), skills (named, encapsulated prompt files for specific tasks like PRD drafting, competitor analysis, ticket creation), sub-agents used as reviewer personas (engineering, design, executive, legal).
  - **Knowledge/context** at three levels — project, user, conversation — sourced from existing KBs but restructured for LLMs. Includes business info, strategy/OKRs, stakeholder metadata, templates, writing style, structured registries (feature index, data catalog, metrics catalog), and user-specific context including a self-learning loop that updates skills after interactions.
  - **Tools/integrations** via MCP to Jira, Figma, Confluence, Power BI, Teams, SharePoint, GitHub, custom data sources.
  - **Triggers**: manual, scheduled, or event-driven from external workflows.
- Five PM use cases walked through:
  1. PRD drafting (demoed live) + multi-reviewer sub-agent critique.
  2. Prototyping grounded in Figma design tokens so output resembles the real product (e.g., GitHub Spark).
  3. Release notes auto-generated from a closed Jira release.
  4. Customer-insight triage that can auto-create Jira tickets and even draft PRs for minor cases (pattern reportedly used at Cursor/OpenAI).
  5. Monthly product analytics (e.g., activation) driven by a skill that runs templated SQL via Power BI MCP.
- Pitfalls flagged: hallucinations (instruct models to refuse ungrounded outputs), garbage-in/garbage-out, context decay (someone must own upkeep), autonomy trap (keep human-in-the-loop, especially for PM decisions), and treating first-shot outputs as final.
- Chat question on skill-file format: only the MD frontmatter (name, description, auto-invoke/slash-command flags) is convention; the rest is prompt-engineering best practice — define required context, structured workflow, avoid noise, and iterate.
