## TL;DR
- Team reviewed a spreadsheet of scheduling-product features (built for Bosch) to classify each as Bosch-specific, reusable, or QOpt/NVIDIA-specific — goal: judge what is a reusable accelerator vs. custom work for the next client / for Oracle-facing messaging.
- Consensus: most "business logic" (allocation rules, constraints, KPIs, demand prediction) is currently hardcoded and will need per-client custom work, but existing code is still an accelerator vs. building from scratch; UI and approval workflow are the most reusable — but only for clients that plan on Work Zones.
- The whole model splits along one fault line: Work-Zone-based planning (what Bosch does) vs. distance-based planning without zones — the latter is essentially a different product ("to be done in the future").

## Key points
- **Work Zones / stability:** Bosch assigns technicians to zones very rigidly (year-ahead, "as stable as possible") with only temporary reassignments; this stability is baked into the algorithm. Could theoretically become a configurable "slider," but current rigidity is atypical.
- **Planning horizon:** Currently max 2–4 weeks; other clients may need a day or a year. Hard to generalize due to performance (horizon scales non-linearly). RosterMod builds a one-week schedule then copies it forward.
- **Allocation rules:** Many rules (neighboring/adjacency logic, busy-zone prioritization, max 3 zones/day) are hardcoded. Much is "common sense," not truly Bosch-specific, but urgent/argent visits were dropped (out of scope) and rule sets vary country-to-country. Ideal future: a configurable file where rules are declared.
- **Max load:** Currently fixed "7 visits/day," ignoring hours, travel time, and traffic. Hours-based logic and travel-time cost are not implemented at all — a major change likely requiring more hardware.
- **QOpt / NVIDIA:** Not special — a classic optimization problem also solvable in plain Python on CPU (already used for local tests). The algorithm only takes a matrix + penalty functions (written by the team); "hours"/"days" concepts are all encoded by the team. Swapping NVIDIA out is easy; a genuinely different algorithm with different inputs would require rework.
- **Demand:** Uses historical demand (1–3 years of files) to forecast where technicians are needed, because current booked visits are too sparse. Speaker C sees this as a "crutch" — prediction should be a separate module (possibly not QOpt), and better forecasting methods exist. Could be reused only if the next client also plans on historical data; ideally prediction is externalized (client provides any forecast).
- **Optimization:** Objectives (productivity, waiting time, workload balance) aren't optimized directly — they emerge from tuned rule penalties/rewards. Future vision: user-added weighted parameters with hard limits (e.g., max 8h/day) vs. soft-weighted preferences.
- **Constraints & skills:** Hard vs. soft constraints should live in a configurable file. Skills logic (technician ↔ visit ↔ zone properties must match) generalizes fairly well.
- **Approval workflow:** Simple, reusable — dispatcher sees QOpt vs. Bosch allocation, approves/denies at zone level with comments (used to track agreement rate; future: regenerate denied items). Currently only zone-level (POC), could be refined to technician/day/visit level.
- **KPIs:** Titles (rows 25–28) are universal, but calculation formulas differ per company; hardcoded now. Clients unlikely to change their formulas → this is custom work, not a UI toggle.
- **UI:** Fairly universal (maps + tables, RosterMod & VisitMod views) — but tied to Work Zones; without zones it must be redrawn. Upload is a workaround (no integration); file formats will differ per client → needs a translation layer or per-client customization.
- **Oracle package usage:** The team believes they used almost nothing from Oracle's provided package — likely only a maps library and Oracle Cloud deploy for easy parts. Their UI/frontend wasn't usable (no Python; the package was built around a route optimizer).
