# Context engineering & harness engineering (Karpathy/Hashimoto framing) — research brief

_question: AI engineering has moved through three eras: prompt → context → harness (Karpathy/Hashimoto framing). Scope of each, divide, tooling, best practices. Why "moved" if context is still active. Single go-to topology resource._
_date: 2026-06-18 · sources: 8 · provenance: (chat)_

## TL;DR

Karpathy coined "context engineering" (X post, June 25 2025) to name the discipline of filling the context window with exactly the right information; Hashimoto coined "harness engineering" (blog post, Feb 5 2026) to name the discipline of building the programmatic wrapper that prevents an agent from repeating mistakes. "Moved" means centre of gravity shifted — the prior techniques remain valid but are now sub-problems of a wider stack. Anthropic has published the clearest canonical taxonomy for each: "Effective context engineering for AI agents" (Sep 29 2025) for context; "Effective harnesses for long-running agents" + "Harness design for long-running apps" (Mar 24 2026) for harness. The `awesome-harness-engineering` GitHub list (ai-boost) is the most complete single-source topology of the harness layer.

## What's known

1. **Karpathy's exact words — context engineering.** [Fact / X post, @karpathy, Jun 25 2025, id 1937902205765607626]: "+1 for 'context engineering' over 'prompt engineering'. People associate prompts with short task descriptions you'd give an LLM in your day-to-day use. When in every industrial-strength LLM app, context engineering is the delicate art and science of filling the context window with just the right information for the next step." In a follow-up same thread: "context engineering is just one small piece of an emerging thick layer of non-trivial software that coordinates individual LLM calls (and a lot more) into full LLM apps" — and listed what that thick layer must do: break up problems into control flows, pack context windows, dispatch calls to LLMs of the right capability, handle generation-verification UI/UX flows, guardrails, security, evals, parallelism, prefetching. [Fact / @karpathy thread, Jun 25 2025]

2. **Hashimoto's exact words — harness engineering.** [Fact / mitchellh.com, "My AI Adoption Journey", Feb 5 2026]: Section titled "Engineer the Harness" — anytime an agent makes a mistake, engineer a solution so that specific mistake is structurally impossible to repeat. His formula: **Agent = Model + Harness**. The harness is the accumulated set of constraints, rules, and environment settings (e.g. AGENTS.md files) encoding every past failure into permanent structure. Published "fully written by hand, in my own words."

3. **The three-era framing is community synthesis, not a joint paper.** [Inference / multiple practitioner sources]: Neither Karpathy nor Hashimoto wrote a "three eras" manifesto together. The framing prompt → context → harness was assembled by the community. Karpathy coined context engineering; Hashimoto coined harness engineering eight months later. The "three eras" label is practitioner consensus, not primary-source.

4. **Context engineering scope — what's included.** [Fact / Anthropic engineering blog, Sep 29 2025; LangChain blog]: Context = all tokens in the context window at inference time. Techniques group into four strategies (LangChain): **Write** (persist state externally — memory files, notes), **Select** (retrieve what's relevant — RAG, semantic search, reranking), **Compress** (summarize/compact — context editing, window management), **Isolate** (separate contexts per sub-agent). Canonical components: system prompt, user turn, few-shot examples, retrieved documents, tool definitions, conversation history/summaries, tool call outputs, structured memory files. Anthropic's internal evals: context editing alone = +29% performance; context editing + memory tool = +39%; 84% token reduction in 100-turn web search eval.

5. **Harness engineering scope — what's included.** [Fact / Hashimoto Feb 2026; Anthropic "Effective harnesses" + "Harness design"; awesome-harness-engineering taxonomy]: The harness is the programmatic wrapper around the model. Components: **(a) Agent loop** — ReAct cycle, turn-based observation, state persistence, extended thinking middleware; **(b) Task decomposition** — plan.md artifacts, milestone-based sprints, tree search over trajectories; **(c) Tool design** — schema conventions, error signaling, structured output, risk annotations, parallel execution; **(d) Permissions & authorization** — five-layer evaluation, headless modes, on-behalf-of delegation; **(e) Memory & state** — session persistence, cross-turn recovery, knowledge consolidation; **(f) Skills / MCP** — Model Context Protocol, A2A protocols, service discovery; **(g) Evals & verification** — CI integration, regression detection, unit-test-style evals; **(h) Observability** — event logging, hook pipelines, cost attribution; **(i) Human-in-the-loop** — approval gating, interruption recovery. Anthropic's Brain/Hands/Session model: Brain = model + loop; Hands = sandboxed execution environments; Session = append-only event log.

6. **The divide: context vs harness.** [Inference / synthesized from Karpathy thread + Hashimoto + Anthropic]: Context engineering = **what the model sees** (token content, structure, relevance). Harness engineering = **what the model operates within** (loop logic, tools, constraints, permissions, evals, orchestration). Context engineering is one component inside the harness. A harness engineer designs the system; a context engineer optimizes the information fed into each inference call within that system.

7. **Why "moved" — is prompt engineering dead?** [Practitioner consensus / Karpathy tweet; Tobi Lütke, Shopify CEO, preceded Karpathy by ~1 week in June 2025]: Not dead — it's subsumed. Prompt engineering techniques (chain-of-thought, few-shot, role instructions) remain valid; they are now micro-optimizations within context engineering, which is itself a sub-discipline within harness engineering. "Moved" = centre of gravity. The practitioner's highest-leverage work shifted from crafting individual prompts to (a) assembling context pipelines, then to (b) designing the full agent environment. Each era's techniques are still in active use.

8. **Canonical topology resources.** [Fact / sources]:
   - **Context engineering:** Anthropic "Effective context engineering for AI agents" (anthropic.com/engineering, Sep 29 2025) — primary, authoritative, includes performance numbers. arxiv:2507.13334 "A Survey of Context Engineering for Large Language Models" — academic taxonomy (Components: retrieval & generation, processing, management; Implementations).
   - **Harness engineering:** `github.com/ai-boost/awesome-harness-engineering` — most complete single-source topology (Design Primitives, Security, Evals, Templates). Anthropic "Effective harnesses for long-running agents" + "Harness design for long-running apps" (Mar 24 2026) — primary practitioner case studies. Hashimoto's "My AI Adoption Journey" (mitchellh.com, Feb 5 2026) — origin source.
   - **Both together:** `github.com/davidkimai/Context-Engineering` — handbook covering context techniques + multi-agent orchestration (1,140+ commits). `arxiv:2507.13334` covers context exhaustively; `awesome-harness-engineering` covers harness exhaustively — no single resource maps both in equal depth.
