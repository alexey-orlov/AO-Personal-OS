## The throughline
Emergent and Brex independently converged on the same foundational discipline: production infra is the moat, not better prompts alone. Emergent built deep proprietary orchestration, memory, and RL infrastructure to ship finished, deployable software instead of brittle demos; Brex enforces human-review chokepoints at every output boundary via proxy-based policy enforcement (crab-trap: ~98% auto-approve / ~2% flagged); and platform-scale governance shows up at GitHub (17M monthly agent PRs met with agentic code-review + CI enforcement so PRs "arrive reviewable") and Snowflake (a centralized AI engineering org certifying skills, running a companywide control plane, and managing a ~20-person team to prevent duplicative or unsafe behavior). The harness-engineering discipline that used to sit alongside this material — Conductor's architecture boundaries, the Sentry bug-triage harness, Codex-as-local-project-environment — now has its own theme: [Agent harness engineering](agent-harness-engineering.md). The evaluation layer is the most durable investment here: eval suites formalize "what success looks like" as scored datasets so models can autonomously explore the "how"; building data→eval pipelines and solid CI comes before any other agent tooling (without automated feedback loops you cannot safely scale agent-driven workflows). A distinct failure mode stands in counterpoint: guardrail accumulation. Layering exception rules past ~14 in a single mega-prompt causes the model to treat every input as an exception (305 submissions, 88 outright failures, 53% Fs among the remainder), requiring a rebuild — guardrails are technical debt requiring parsimony and active measurement, not progressive hardening. The quality complement: positive curation engineering (Ploy: 3,500 targeted prompts + human-curated lookbooks) is as load-bearing as guardrail parsimony. Advanced verification instruments now include streaming RAG (begin retrieval mid-utterance: ~1.5s latency reduction for voice agents), Lean-based formal verification (machine-checkable proofs as a concrete defense against hallucination), and the Mozilla security harness (programmatic tools + dedicated verifier sub-agent → hundreds of verified Firefox fixes including ~15-year-old bugs; pre-run LLM scoring by attack surface + memory-safety likelihood directs compute before any agent is deployed). Agents also run far more rigorous, continuous experiments than any staff engineer can: Bloom filter index found after a week of agent-driven benchmarking; matrix-style testing across every open-source column store under production conditions surfaces overlooked trade-offs. Environment-style training — models operating APIs, servers, and document corpora — improves coding ability even without explicit programming access, suggesting tool-use patterns are genuinely transferable. Inference cost is a first-class concern throughout: Brex's Magpie attributes every token-spend dollar to products and customers, finding that token-maxing regions correlate with faster revenue growth while unattributed mandates (enterprises declaring AI mandatory, measuring employees on usage without instrumentation) drive waste and governance debt that CIOs must now address as a C-suite priority. A parallel commercial frontier adds four practical thresholds: stair-step to autonomy (SaaStr 10K: start as a dashboard, add one workflow at a time, 15 min to launch / months to full autonomy); action-bias over data-perfection (Salesforce SDR: 70% deliverability in days, not months — web agents trained on site/FAQ data sidestep CRM hygiene); headless composable surfaces (products without APIs get replaced, not integrated); and purpose-built compute (Vercel Fluid: traditional request/response cloud breaks under long-running tool-calling agents — event-triggered mini-servers cut costs up to 85%). The semantic data layer is the common load-bearing piece: without grounded SQL translation the agent hallucinates; with it, analytics go from one-week tickets to under one minute (Vercel D0).

### Start retrieval while the user is speaking to cut voice-agent latency
Streaming RAG designs let a retrieval-augmented model begin fetching documents from partial transcriptions so the system can answer conversational voice queries with low delay rather than waiting for full utterances. The paper evaluates fixed-interval streaming and a learned trigger that decides when partial-query retrieval is worth it, and reports latency reductions of roughly 0.5 seconds on synthetic datasets and ~1.5 seconds on human speech without dropping retrieval accuracy. Practically, this shows you can keep RAG-style factual grounding for voice agents while achieving conversational responsiveness by deciding when and how to run retrieval on partial input.

### Formal verification (Lean) is becoming a practical path to verified models and code
The community is integrating LLMs with Lean and other interactive theorem provers to produce machine-checkable proofs and verified code; progress is rapid enough that models are tackling IMO-level problems and large formal libraries (mathlib) provide substantial corpora. Work discussed includes using Lean as both a theorem prover and a functional programming/spec language (and even a Torch-for-Lean stack), enabling proofs about neural nets (e.g., flash-attention equivalence) and certified properties of numerics/robustness. This matters because it gives a tractable route from LLM outputs to verifiable correctness for math, scientific claims and program behavior — a concrete defense against hallucination and silent bugs.

### Ship with many agentic workers: treat software work like a real-time strategy game
Channel AI advocates running many lightweight agent workers in parallel (spawned from an orchestrator), pushing each task as far as possible toward a PR, then using high-visibility monitoring, rich knowledge-bases, and rapid human corrections — prioritizing throughput and satisficing over single-threaded perfection. ### Eval suites are the modern PRD for AI-driven features
Evals formalize 'what success looks like' the way PRDs used to, by encoding user examples and quantitative scoring so models can autonomously explore the 'how.' In practice they built datasets of doc-site questions, used models to generate and then refine scoring functions (e.g., concise code snippets, single-language answers), and applied those scorers to rank outputs automatically. This matters because it converts subjective product taste into repeatable, measurable evals—then human vibe checks (they iterate with a designer named David) are used only to refine the scorer rather than manually reviewing every example.

### Their copy-edit benchmark optimized an unvalidated target
An internal copy-editing benchmark spent five weeks and 180 agent threads optimizing for strict recall against one editor's historical edits, targeting 70% match, before anyone validated that target was achievable. Fable's own review of the project called this out: the team's best runs plateaued around 47.5–50%, which may already be near the human ceiling (even the target editor is inconsistent with her own past edits), so the effort went into an ill-posed objective instead of cheaper, higher-information fixes.

### Switch evaluation from exact recall to acceptance precision
Because humans (even the target editor whose past edits were being matched) are inconsistent, the fix is to flip the objective from exact historical-recall to acceptance precision in a live suggestion loop — have the model produce edits and collect binary accept/reject judgments from a human to build a label factory. One human spending two hours judging ~60 cases is far cheaper than the five-week benchmark effort and yields training data aligned to whether an editor would actually accept an edit, not whether it matches a noisy past.

### Too many guardrails can break an AI agent
Layering many exception rules into a single mega-prompt caused the deck-grading agent to reject most submissions — of 305 recent submissions 88 failed outright and of the 216 that worked 53% received Fs. After about the 14th guardrail the system considered everything an exception and stopped functioning, forcing a rebuild and removal of most rules. The non-obvious lesson: guardrails are technical debt — over-constraining an LLM-driven flow throttles useful outputs and requires careful measurement and parsimony.

### Giving models real tools produces reproducible, actionable bug reports
Mozilla built a harness that gives an LLM programmatic tools — e.g., run bash, open a browser, build Firefox, and feed test HTML to a fuzzer — so the agent can turn code hypotheses into executable test cases. Those artifacts are run on a fuzzing build (with address sanitizer) and a verifier sub-agent confirms crashes, which eliminates the flood of unactionable AI reports and produces reports engineers can act on. The approach let Mozilla escalate to hundreds of high-quality fixes because the agent output was already verified or easily verifiable.

### Relentless agent loops find edge cases humans would miss or tire of
Agents can exhaust many hypotheses without cognitive decline: Mozilla's agents repeatedly generated and ran dozens of variants until one hit — an example being a 'legend' element bug that required 14 tries before the agent produced a crash-reproducing HTML. That persistence matters because many security bugs are subtle, require long chains of reasoning from web input to C++ implementation, and are expensive for humans to brute-force manually. The result is discovery of very old or obscure vulnerabilities (including a ~15-year-old XSLT bug) that human triage often misses.

### Prioritize targets before deploying agents; LLM-based scoring directs compute
Firefox's codebase is huge, so Mozilla uses lightweight LLM scoring to prioritize files by two key axes: likelihood of memory-safety issues and accessibility from a web page (i.e., attack surface). The scorecard uses file types (C++, WebIDL, etc.), history (how often a file has been scanned/found bugs), and other heuristics to allocate agent runs and compute efficiently, and they also consider commit scanning or performance benchmarks as alternate inputs. Prioritization reduces wasted compute and human review effort and lets teams mobilize around the highest-impact findings.

### Patching agent generates draft fixes and re-verifies; humans still gate landing
Mozilla added a patching agent that generates proposed fixes, applies them to a build, and re-verifies that the crash or issue goes away; in cases like nsULContentSync the agent suggested a simple assert change that became the landed fix after human review.
